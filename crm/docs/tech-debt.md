# Технический долг и планы развития

## 1. Telegram Worker Bot (Интеграция)

### Описание
Создание Telegram-бота, дублирующего функциональность мобильного приложения Worker App (worker-app).

### Функционал
- Авторизация сотрудника через табельный номер.
- Просмотр списка активных задач (`/api/mobile/tasks`).
- Ввод выработки (количество, брак) через диалог с ботом.
- Уведомления для мастеров о необходимости подтверждения записей.

### Преимущества
- Push-уведомления в реальном времени.
- Не требует установки отдельного приложения.
- Привычный интерфейс для работников.

### Сложности
- Ввод данных с множеством полей (например, настил: 7 полей) неудобен в чате.
- Сложная визуализация таблиц и графиков.

### Статус
📋 **Запланировано**

## 4. Нагрузочное тестирование Worker + CRM (200–300 одновременных пользователей)

### Описание
Нет формализованного нагрузочного профиля и критериев приемки для сценария массовой работы сотрудников в `worker-app` с одновременной синхронизацией данных в CRM.

### Риски
- Деградация API при пиковых сменах.
- Рост задержки синхронизации Worker → CRM.
- Нарушение консистентности данных (`task_entries` vs `operation_entries`).
- Непредсказуемое поведение rate-limit/realtime под нагрузкой.

### План реализации
1. Подготовить staging, максимально близкий к production (конфиг Supabase, RLS, индексы, лимиты).
2. Добавить сценарии в `k6`/`Locust`:
   - `steady`: 200–300 одновременных пользователей, 20–30 мин.
   - `spike`: резкий рост 50 → 300 за 1–2 мин.
   - `soak`: 150–200 пользователей, 2–4 часа.
3. Зафиксировать SLO:
   - API `p95` < 800 ms для ключевых роутов.
   - Ошибки `5xx` < 1%, контролируемый `429` отдельно.
   - Задержка синхронизации Worker → CRM `p95` < 3 сек.
4. Ввести `trace_id/request_id` для измерения `t_write` (worker) и `t_seen` (CRM/realtime).
5. После каждого прогона выполнять SQL-проверки консистентности:
   - нет “осиротевших” связей;
   - `task_entries` и `operation_entries` согласованы по ключевым полям/статусам.
6. Сформировать единый отчет прогона (метрики, ошибки, узкие места, рекомендации).

### Критерии приемки
- Все профили пройдены без критических деградаций.
- SLO выполнены по `p95`, error rate и sync delay.
- SQL-проверки консистентности проходят без нарушений.
- Отчет сохранен и согласован командой.

### Статус
🟡 **Высокий приоритет**

## 2. Разделение данных: task_entries vs operation_entries

### Описание
17 API роутов читают данные из устаревшей таблицы `public.operation_entries`, в то время как новые записи пишутся в `shveyka.task_entries`.

### Влияние
- Неправильный расчет зарплаты.
- Пустая аналитика на дашборде.
- Некорректные ответы AI-ассистента.

### План решения
Полный перевод всех API роутов на чтение из `shveyka.task_entries`. (См. `docs/adr/001-task-entries-vs-operation-entries.md`).

### Статус
🔴 **Критично** (Ожидает реализации)

## 3. Доступ роли Production Head

### Описание
Роль `production_head` не имеет доступа к пунктам меню "Замовлення", "Партії" и другим критичным разделам в Sidebar, хотя логически должна ими управлять.

### План решения
Добавить `production_head` в массивы `roles` для соответствующих пунктов меню в `Sidebar.tsx` и разрешающие роли в API роутах.

### Статус
🟡 **Средний приоритет**
