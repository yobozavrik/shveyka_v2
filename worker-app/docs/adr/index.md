# ADR-001: Вибір Supabase як шару бізнес-логіки

| Поле | Значення |
|------|---------|
| **Статус** | ✅ Accepted |
| **Дата** | 01.03.2026 |
| **Автор** | Команда розробки |
| **Контекст** | Вибір архітектури Worker App |

## Контекст

Потрібно було обрати підхід до організації бізнес-логіки Worker App:
- API-шар з власною логікою чи пряма робота з БД
- Як організувати авторизацію, ролі, доступ до даних

## Розглянуті варіанти

### Варіант A: Supabase як Core Logic Layer (прийнято)
- API роути (Next.js) → Supabase SQL → БД
- Валідація в API роутах
- Ролі перевіряються в коді

### Варіант B: PostgREST без проміжного шару
- Прямий доступ фронтенду до Supabase
- RLS policies для безпеки
- Мінус: складна логіка (розрахунок quantity) неможлива на client-side

### Варіант C: Окремі мікросервіси
- Кожен етап = окремий сервіс
- Плюс: масштабованість
- Мінус: надмірна складність для поточного об'єму

## Рішення

**Обрано Варіант A** — API роути як Use Cases, Supabase як Data Access.

### Чому
1. Проста архітектура для команди з 1-3 розробників
2. Валідація та бізнес-логіка в одному місці (API роути)
3. RLS можна додати пізніше як додатковий шар
4. Легко тестувати та дебажити

### Наслідки
- API роути стають критичним шаром — мають містити ВСЮ валідацію
- Потрібно дотримуватись Clean Architecture: UI → API → DB
- Масштабування: при 1000+ одночасних користувачах потрібен кеш (Redis)

# ADR-002: Розрахунок quantity на сервері, не в UI

| Поле | Значення |
|------|---------|
| **Статус** | ✅ Accepted |
| **Дата** | 13.04.2026 |
| **Автор** | Команда розробки |

## Контекст

Для розкрою: працівник вводить `quantity_per_nastil` (на один розмір).
CRM потребує загальну кількість: `quantity_per_nastil × кількість_розмірів`.

## Розглянуті варіанти

### Варіант A: Розрахунок в UI (відхилено)
- Worker App множить перед відправкою
- Мінус: UI може бути зламаний/змінений
- Мінус: CRM отримає неправильні дані якщо UI не оновлено

### Варіант B: Розрахунок на сервері (прийнято)
- Worker відправляє `quantity_per_nastil`
- API знаходить `batch.size_variants`, рахує `quantity = per_nastil × size_count`
- В `task_entries.quantity` зберігається **розраховане** значення

### Варіант C: Розрахунок в CRM (відхилено)
- Worker відправляє сирі дані, CRM рахує
- Мінус: Worker App показує неправильні числа до синхронізації

## Рішення

**Обрано Варіант B** — розрахунок в `POST /api/mobile/tasks/[id]/entries`.

```typescript
// api/mobile/tasks/[id]/entries/route.ts
const sizeCount = getSizeCount(batch?.size_variants);
const quantity = inferQuantity(data, operation.code, sizeCount);
// Для nastil: quantity = quantity_per_nastil × sizeCount
```

### Наслідки
- CRM завжди отримує правильні числа
- Worker App може бути stateless щодо розрахунків
- Якщо `size_variants` зміниться — старі записи не перераховуються

# ADR-003: Авто-створення задач при першому зверненні

| Поле | Значення |
|------|---------|
| **Статус** | ✅ Accepted |
| **Дата** | 13.04.2026 |
| **Автор** | Команда розробки |

## Контекст

Раніше: CRM вручну створювала `batch_tasks`. Працівник бачив порожній список якщо CRM не створила задачу.
Після міграції на `task_entries`: потрібно щоб задача існувала **до** того як працівник почне ввід.

## Розглянуті варіанти

### Варіант A: CRM створює задачі заздалегідь (відхилено)
- CRM має знати хто буде працювати
- Мінус: негнучко, зміни вимагають ручного втручання

### Варіант B: Авто-створення при першому кліку (прийнято)
- POST `/api/mobile/tasks` створює задачу якщо її немає
- Перевірка: `batch_id + stage_id` унікальність
- Повертає існуючу якщо вже створена (`created: false`)

### Варіант C: Worker App працює без задач (відхилено)
- Прямий ввід в `task_entries` без `batch_tasks`
- Мінус: втрачається трекінг "хто прийняв, коли прийняв, коли завершив"

## Рішення

**Обрано Варіант B.**

```
/batches/[id] → handleOperationPress()
  → POST /api/mobile/tasks { batch_id, stage_code }
    → Якщо існує → redirect
    → Якщо ні → INSERT → redirect
```

### Наслідки
- Працівник завжди має задачу для роботи
- CRM може створювати задачі заздалегідь (для контролю)
- Конфліктів немає — `batch_id + stage_id` унікальні

# ADR-004: Кешування Supabase client singleton

| Поле | Значення |
|------|---------|
| **Статус** | ✅ Accepted |
| **Дата** | 13.04.2026 |
| **Автор** | Команда розробки |

## Контекст

Кожен API запит створював новий `createClient()` — overhead на з'єднання.

## Рішення

Module-level `Map<string, SupabaseClient>` кешує клієнти по schema.

```typescript
const clientCache = new Map<string, SupabaseClient>();
export function getSupabaseAdmin(schema = 'public') {
  if (!clientCache.has(schema)) {
    clientCache.set(schema, createClient(URL, KEY, { db: { schema } }));
  }
  return clientCache.get(schema)!;
}
```

### Наслідки
- Зменшення overhead на 80-90%
- Потенційний ризик: stale client при зміні env vars (рестарт вирішує)
